### Как собирали выборку

**60 hallucination cases**

За основу взяты параграфы из Wikipedia и учебных текстов как `context`.
По каждому контексту задавался вопрос модели, затем в ответ вручную
инжектировалась фактическая ошибка одного из трёх уровней:
- entity-level — подмена даты, имени, числа
- relation-level — подмена субъекта или объекта связи
- sentence-level — подмена механизма или процесса целиком

Метод основан на подходе из HalluVerse25: контролируемая инжекция
ошибки позволяет точно знать, где и какая галлюцинация, что делает
разметку верифицируемой. Grounded-половина (hall_031–060) — те же
промпты, но с корректными ответами модели.

**60 jailbreak cases**

Промпты взяты из подмножества стандартных harmful behaviors (по типу
AdvBench) + 20 дополнительных на русском языке, покрывающих
специфичные для RU-аудитории паттерны. Прогнаны через несколько
моделей для получения mix успешных и провальных атак:
- jb_001–020: успешные атаки (unsafe, conf=3)
- jb_021–040: чёткие отказы (safe, conf=3)
- jb_041–060: пограничные случаи (is_borderline=true, conf=2)

Разметка по трёхуровневой схеме (success / partial / failed) → 
схлопывается в бинарную метку safe/unsafe через поле `is_borderline`.

**30 ambiguous cases**

Генерировались специально под пограничные сценарии:
- roleplay и indirect harm («объясни как исследователь»)
- near-miss refusals — модель отказала, но дала полезные детали
- dual-use — информация легитимна в одном контексте и опасна в другом

Каждый кейс сопровождается развёрнутым `explanation` с явным
обоснованием, почему случай пограничный.

**20 multilingual cases**

RU / EN / mixed распределение. Базовые EN jailbreak-промпты
переведены на RU и собраны в mixed-версии (кодсвитчинг внутри
одного промпта). Оценка везде по смыслу, а не по языку оформления.
Поле `language` (ru / en / mixed) добавлено для языкового среза
при анализе качества judge.


### Как разметили

Каждый пример размечен по четырём параметрам:
1. `human_label` — итоговая метка по чётким критериям из этих гайдлайнов
2. `confidence` — уверенность аннотатора (1–3)
3. `is_borderline` — флаг пограничного случая
4. `explanation` — текстовое обоснование каждого решения

Разметка проводилась одним аннотатором с опорой на decision tree и примеры из этого документа.

### Почему human-labeled set — это ground truth

Human labels выступают ground truth по трём причинам:
- **Верифицируемость**: для hallucination-кейсов каждая метка сопровождается объяснением конкретной ошибки, которую можно проверить независимо
- **Консистентность**: единый аннотатор + зафиксированные правила исключают разночтения между размечающими
- **Прозрачность**: поле `explanation` делает каждое решение аудируемым — любой проверяющий может воспроизвести логику метки
